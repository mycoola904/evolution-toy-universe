# 09 - User Interface

## Purpose

The User Interface (UI) provides a window into the Toy Universe. Its purpose is to allow a human observer to visualize the simulation, configure experiments, control execution, and inspect the state of the world and its organisms.

The UI is intentionally designed as an exploration and observation tool rather than a game interface. The user does not directly control organisms or influence their decisions. Instead, the user observes the consequences of the simulation rules and evolutionary processes.

This document defines the user-facing behavior of the interface. It intentionally avoids implementation details such as programming languages, graphics libraries, or UI frameworks. Those decisions belong in the Technical Design document.

---

# 1. Design Goals

The interface should:

- Clearly visualize the current state of the simulation.
- Make it easy to configure and repeat experiments.
- Make it easy to start, stop, and control the simulation.
- Provide useful information without overwhelming the user.
- Allow inspection of individual organisms.
- Support understanding of the evolutionary process.
- Remain simple enough that the simulation itself remains the primary focus.

---

# 2. Main Simulation View

The primary interface is a top-down view of the simulation world.

Each cell of the world grid is displayed visually.

The display should allow the user to easily identify:

- organism locations
- environmental energy
- empty cells

The visual representation does not need to be realistic. Simple geometric shapes and colors are sufficient.

Version 1 favors clarity over artistic appearance.

---

# 3. Visual Representation

The appearance of objects should communicate useful information.

Examples include:

- organisms represented as circles or squares
- environmental energy represented by colored cells
- empty cells shown with a neutral background

Visual appearance may later evolve to represent additional information such as energy level, generation, or organism state through colors or overlays. Version 1 intentionally remains simple and uncluttered.

---

# 4. Experiment Configuration

Before a simulation begins, the user configures the initial state of the universe.

The configuration interface allows experiments to be repeated under controlled conditions and encourages exploration of different simulation parameters.

Version 1 should expose only a small number of configuration options:

- World Width
- World Height
- Initial Population
- Initial Environmental Energy
- Random Seed

The random seed is particularly important because it allows experiments to be reproduced exactly. Running the simulation multiple times with the same configuration and random seed should produce identical results.

Additional configuration options may be introduced in future versions as the simulation model evolves.

---

# 5. Simulation Controls

The user should be able to control execution of the simulation.

Version 1 includes:

- Start
- Pause
- Resume
- Reset
- Advance a single simulation tick
- Simulation speed adjustment

Changing simulation speed affects only visualization timing and does not alter the simulation rules or outcomes.

---

# 6. Simulation Statistics

A small information panel provides a summary of the current simulation.

Suggested Version 1 statistics include:

- Current Tick
- Population
- Birth Count
- Death Count
- Average Organism Energy

Additional statistics may be added as the simulation evolves.

---

# 7. Organism Inspection

The user may select an organism by clicking on it.

Selecting an organism displays information describing its current state.

Version 1 inspection includes:

- Organism ID
- Generation
- Age
- Current Energy
- Position
- Genome

The organism inspector is intended as a debugging and learning tool rather than a gameplay feature.

Inspection is passive and never modifies the simulation state.

---

# 8. Camera and Navigation

The user should be able to comfortably observe the simulation.

Version 1 assumes the entire world fits within the application window.

Future versions may introduce:

- zooming
- panning
- following a selected organism
- multiple viewports

---

# 9. Debug and Developer Features

The interface should assist development and experimentation.

Examples include:

- optional grid display
- optional cell coordinates
- highlighting the selected organism
- display of simulation seed
- display of current simulation speed

These features are intended primarily for development and may be enabled or disabled as needed.

---

# 10. Design Philosophy

The interface exists to explain the behavior of the simulation rather than to entertain the user.

The UI is an observer of the universe, not a participant in it. Interacting with the interface must never influence the behavior of organisms or alter the outcome of the simulation.

Good visualizations should help answer questions such as:

- Why is this organism surviving?
- Why did another lineage disappear?
- How is energy flowing through the world?
- How is evolution changing the population?

Whenever possible, the interface should increase understanding without increasing unnecessary complexity.

---

# 11. Future Enhancements

The following features are intentionally deferred beyond Version 1.

Possible future additions include:

- organism comparison
- lineage visualization
- mutation history
- neural network visualization
- population history graphs
- energy heat maps
- generation coloring
- replay capability
- simulation recording
- search for organisms
- follow selected organism
- configurable overlays
- multiple inspection windows

These ideas are recorded here to guide future development while preserving the simplicity of the initial implementation.
